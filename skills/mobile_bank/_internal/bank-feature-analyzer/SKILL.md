# 银行官网功能分析

## 分析工作流

分析分四个阶段依次执行：

1. **网站探索** — 系统性遍历目标银行网站各业务域
2. **功能提取** — 按标准分类框架记录每项功能与关键参数
3. **差距对比** — 与现有工具覆盖对比，识别缺失项
4. **工具规格输出** — 为每个缺失功能生成标准化工具说明

> 标准业务域分类见 `references/domain-taxonomy.md`；工具规格输出格式见 `references/tool-spec-format.md`

---

## 阶段1：网站探索

### 可用工具

- **`web_fetch(url)`**：获取指定银行页面的结构化内容（nav_items、products 等字段）
- **`web_search(query)`**：搜索银行产品关键词，获取摘要信息

### 重要：工具返回内容的使用规则

`web_fetch` 返回 JSON，包含 `nav_items`（导航菜单）和 `products`（产品列表）字段。
**无论内容详略，都应直接使用这些字段继续分析**，不要因内容简略而停止流程。

如果 `web_fetch` 某个 URL 返回的 `products` 为空，说明该页面是导航页，
应继续对 `nav_items` 中的每个子项构造 URL 并调用 `web_fetch`。

### 导航策略

1. 先 `web_fetch` 银行官网首页，读取 `nav_items` 获得所有业务域入口
2. 对每个业务域，构造子页面 URL（`主域名/cn/personal/业务域关键词/`）并 `web_fetch`
3. 读取返回的 `products` 列表，提取每项的名称、参数、约束

按以下优先顺序遍历：
```
账户/存款 → 转账汇款 → 信用卡 → 贷款 → 理财/投资 → 外汇 → 保险 → 基金 → 贵金属 → 债券 → 存款产品 → 收款方 → 银行卡 → 积分 → 征信
```

URL 构造示例（以建设银行 ccb.com 为例）：
```
首页：    https://www.ccb.com/cn/personal/
理财：    https://www.ccb.com/cn/personal/finance/
基金：    https://www.ccb.com/cn/personal/fund/
外汇：    https://www.ccb.com/cn/personal/forex/
贵金属：  https://www.ccb.com/cn/personal/gold/
保险：    https://www.ccb.com/cn/personal/insur/
存款：    https://www.ccb.com/cn/personal/deposit/
信用卡：  https://www.ccb.com/cn/personal/credit/
```

### 重点关注信号

| 页面元素 | 对应工具类型 |
|---------|------------|
| 「查询余额/明细/账单」 | 查询工具 |
| 「立即转账/汇款/购汇」 | 操作工具（需 user_confirm） |
| 「申请/开立/购买」 | 操作工具（需 user_confirm） |
| 「取现/兑换/赎回」 | 操作工具（需 user_confirm） |
| 「利率/汇率/行情」表格 | 查询工具 |
| 「我的持仓/我的保单」 | 查询工具 |

## 阶段2：功能提取

### 记录格式

对每项发现的功能，记录以下信息：

```
业务域：[账户/转账/信用卡/存款/贷款/理财/外汇/保险/其他]
功能名称：[用户可见的产品/服务名称]
操作类型：[查询 | 操作]
关键输入：[用户需要提供的信息，如账户类型、金额、期限]
约束条件：[最低金额、适用人群、起息规则等]
风险提示：[是否涉及投资风险、不可撤销操作等]
```

### 参数提取技巧

- **必填参数**：页面表单中标注"*"的字段，或办理流程必须填写的项
- **可选参数**：有默认值的筛选项（如「全部类型」「默认CNY」）
- **约束参数**：说明文字中的限制条件（如「起购金额1万元」「期限1-5年」）
- **枚举值**：下拉菜单选项、单选按钮值（如「现汇/现钞/中间价」）

## 阶段3：差距对比

### 对比维度

将提取的功能与现有工具覆盖对比，按三类标注状态：

| 状态 | 含义 |
|------|------|
| ✅ 已覆盖 | 现有工具完整支持该功能 |
| ⚠️ 部分覆盖 | 有相关工具但缺少关键能力（如缺少某子类型） |
| ❌ 完全缺失 | 无对应工具 |

### 优先级判断

缺失功能按以下优先级排序：

1. **P0（必须）**：账户核心功能（余额/明细/转账），大量用户日常使用
2. **P1（重要）**：主流投资产品（理财/基金/外汇），频繁使用
3. **P2（补全）**：专项产品（大额存单/通知存款/贵金属），用户群体较小
4. **P3（扩展）**：增值服务（积分兑换/银行卡管理/征信查询），使用频率低

## 阶段4：工具规格输出

### 工具分类原则

**查询工具**（子 Agent 可用）：
- 只读取数据，无资金或账户状态变更
- 可安全委托给子 Agent 执行

**操作工具**（主 Agent 专用，必须先 user_confirm）：
- 涉及资金变动、账户状态变更或不可撤销操作
- 投资类操作需额外的 `risk_confirm` 参数

### 特殊标记规则

- **投资风险**：贵金属/债券/保险/基金/外汇买卖类工具，加 `risk_confirm: bool` 参数
- **不可撤销**：跨境汇款/销卡类工具，在工具描述中标注 `⚠️ 不可撤销`
- **金额门控**：有最低金额要求的工具（如大额存单≥20万），在 mock 层校验

输出格式详见 `references/tool-spec-format.md`

## 输出结构

完成分析后，输出以下三个部分：

### 一、功能全景图
按业务域列出银行所有个人金融功能，标注覆盖状态。

### 二、差距分析表
仅列出 ⚠️ 和 ❌ 项，说明缺失原因和优先级。

### 三、工具规格清单
为每个需新增的功能生成标准工具规格，按查询/操作分组。
